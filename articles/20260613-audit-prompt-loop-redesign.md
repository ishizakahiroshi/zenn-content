直近の開発は、9割をAIエージェントに任せています。その流れで少し前に、AIにコードを監査させる——セキュリティの穴や脆弱性、バグを洗い出させる——ためのプロンプト集を、Claude Code と Codex CLI 向けに公開しました。

そのうちの1つ、最新モデル Claude Fable 5 向けのプロンプトを、今回ごっそり書き直しました。AIにコードレビューやセキュリティ監査をやらせたい人に、何かしら刺さればと思って書いています。

きっかけは、自分の作りが間違っていたと気づいたことです。ひとことで言うと、指示を書きすぎていました。

## 前の版は、手取り足取りだった

監査プロンプトというのは、AIに「このコードを、こういう観点で調べて、見つけた問題をこう確かめて、こう直して」とお願いする長い文章です。

前に作った Fable 5 版は、その手順をかなり細かく書き込んでいました。調べる観点を並べて、確かめ方を並べて、注意点を並べて。丁寧に書けば書くほど精度が上がるはず、と思っていたんです。

実際それで動くし、悪くはなかった。ただ、もっと良くできる余地があったことに、あとから気づきます。

## 「細かく指示するな」と書いてあった

ある日、Fable 5 を作っている側の人が書いた、このモデルの使い方の記事を読みました。そこに、こうありました。

Fable 5 は、人間が手取り足取り指示するより、自分で間違いに気づいて直せる「ループ」を用意してやった方が、良い結果を出す。

ぐさっときました。自分がやっていたのは、まさにその逆だったので。

細かく手順を書くほど、モデルは「言われた通り」に動こうとして、自分で深く考える余地が減る。良かれと思った丁寧さが、たぶん足を引っ張っていた。

![](./images/2026-06-13_ai-audit-prompts-loop_fig1.png)

ここから、直し方の方針が決まりました。手順で縛るのをやめて、自分で気づいて直せる仕組みの方を作る。やったのは、大きく3つです。

## 一番効いたのは、検証役を別人にしたこと

手応えが一番あったのは、ここでした。

前の版では、コードを調べたエージェントが、自分の見つけた問題を自分で「これは本当に正しいか?」と確かめていました。一人二役です。

でも、自分の出した答えのアラは、自分ではなかなか見えません。テスト前に自分で答え合わせをして、間違いに気づけないのと同じです。

そこで、確かめる作業だけを別のエージェントに渡しました。問題の中身と根拠だけを渡して、まっさらな状態で「これ、本当に成立する?」と疑ってもらう。調べた本人の思い込みが入らない分、あやしいものがちゃんとふるい落とされます。

これは効きました。

![](./images/2026-06-13_ai-audit-prompts-loop_fig2.png)

## 終わりの条件を、チェックリストから採点表に

2つ目は、「どうなったら完了か」の決め方です。

前は、完了の条件をただ箇条書きで並べていました。AIは、だいたいできたところで「終わりました」と言いがちです。

なので、完了条件を採点表の形にして、全部の項目が満たされたと別のエージェントが判定するまで終われない、というループにしました。中途半端なところで切り上げさせない、という仕掛けです。

## 作業メモを、ただのログから「学ぶ記憶」に

3つ目は、作業中のメモの取り方です。

前は、やったことを書き留めるだけのログでした。今回は、流れを変えました。間違えたら理由も残す。原因を調べたら、推測のままにせず事実として確定させる。それを「このコードではこういう決まりだ」という一般ルールに昇華して、次に同じことを調べ直さない。

失敗を、ちゃんと次に活かす形にした、という感じです。

そして最後に、引き算もしました。細かい手順は削って、ゴールと「やってはいけないこと」だけを渡す。進め方そのものは、モデルに任せる。書きすぎを、やめました。

## ついでに、ファイル名から「5」を外した

これは小さな話ですが。

前はファイル名を claude_fable5 のように「5」付きにしていました。でも Fable はこの先、6, 7 と上がっていくはずです。そのたびに名前が古びて、新しいモデルが対象外だと誤解されるのも嫌だなと。

なので、バージョン番号を外して claude_fable に統一しました。次のモデルが出ても、名前はそのまま使い続けられます。中身に「例: Fable 5」とだけ添えておく形に。

## 使い方は、変わらず簡単です

Claude Code でモデルを Fable に切り替えて、リポジトリに置いたプロンプトを「これで監査して」と頼むだけです。

強度やスコープ、調べる観点を引数で絞ることもできますが、全部省略してもデフォルトで動きます。まず全体をざっと見たいならそのまま、認可まわりだけ深く見たいなら観点を絞る、という使い分けです。

ビルドもコミットも本番DBへの操作もしない、止まらず最後まで走り切る、迷ったら記録して次へ進む——この約束ごとは、前の版から変えていません。安心して任せられる範囲は守ったまま、中身の賢さだけ上げたつもりです。

## 同じことを、他のプロンプトにはやらなかった

ここまで読むと、「じゃあ他のも全部そうやって削ればいい」と思うかもしれません。でも、やりませんでした。

このプロンプト集には、Fable 以外にも、複数のAIを並列で走らせる版や、別のAIツール向けの版があります。そっちは、わざと細かく指示を書いたまま残しています。

並列で走らせる版は、調べる観点や手がかりを具体的に並べておかないと、各AIの担当があいまいになって取りこぼす。別ツール向けの版は、細かく列挙した方が素直に動く。細かさが効くかどうかは、相手のモデルや動かし方で変わるんです。

「指示は削るのが正解」と一般化したくなったけど、それは Fable に限った話でした。相手によって、ちょうどいい伝え方は変わる。当たり前のことを、作り直しながら思い出しました。

ただ、どの版にも共通で1つだけ足したルールがあります。「自分で確かめていないことを、できたことにして報告しない」。これはモデルを問わず効くはずです。

![](./images/2026-06-13_ai-audit-prompts-loop_fig3.png)

## 締め

正直に言うと、今回の直しは、新しい機能を足したわけではありません。むしろ、自分が書きすぎた指示を削る、引き算の作業でした。

AIに任せる、というのは、丁寧に指示を書き込むことだと思っていました。でも、賢いモデルほど、細かく指図するより、間違いに気づける環境を用意して、あとは任せる方がいいのかもしれない。

自分で書いた指示を、自分で削る。ちょっと寂しい作業でしたが、たぶんこっちが正解です。小さく。書きすぎないことを、これから覚えていきます。

## 追記：直した矢先に、Fable が止まった

ここまで書き終えた、まさにその日のことです。Fable 5 が止まりました。

アメリカ政府の輸出管理の指示で、外国籍の利用が一律ストップ。日本にいる自分も、いまは Fable 5 を呼べません。安全性をかなり上げたはずのモデルに「抜け道があるらしい」という指摘が引き金だったとか。詳しいことは、正直まだよく分かっていません。

Fable 5 のために気合いを入れて作り直したプロンプトが、肝心の Fable 5 を呼べない。タイミングが良すぎて、思わず笑ってしまいました。

ただ、今回直した中身——検証を別の役に任せる、終わりを採点表で決める、メモを学ぶ記憶にする——は、どのモデルでも効く「ループの設計」の話です。当面は手元の別のモデルで同じ考え方を使いながら、Fable が戻ってきたら、このプロンプトをそのまま投げるつもりです。

戻ってきますように。

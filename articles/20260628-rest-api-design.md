## 先に結論だけ

X で流れてきたバックエンド面接問題が出発点だ（出典: [@jahirsheikh8 のポスト](https://x.com/jahirsheikh8/status/2070849696730607695)）。

> 次の API 設計のうち、**はるかにスケーラブルなもの**が一つあります。どちらを選びますか？

**A 案（パスで分ける）**

```
GET /api/v1/users/profile
GET /api/v1/users/orders
GET /api/v1/users/settings
```

**B 案（クエリで分ける）**

```
GET /api/v1/users?type=profile
GET /api/v1/users?type=orders
GET /api/v1/users?type=settings
```

### 答え：A でも B でも、即答した時点で不正解

`profile` / `orders` / `settings` が「別物（別リソース）」なのか「同じものの出し分け」なのか。**前提が示されていない**。前提次第で正解はひっくり返る。

| 前提 | 正解 | 理由（一行） |
| --- | --- | --- |
| これらが**別リソース**（型・権限・ライフサイクルが違う） | **A（パス）** | 別物はパスで分ける。クエリで束ねると型崩壊・ポリシー分離不能 |
| これらが**同一スキーマの出し分け**（`type` は単なる条件） | **B（クエリ）** | 同じものの絞り込みはクエリ。拡張・合成に強い |

判断軸はたった一行。

> **パスは「別のリソース」、クエリは「同一リソースの絞り込み」。**

だから満点回答はこうなる。

> 「異なるリソースならパスで分ける A、同一スキーマの出し分けならクエリの B。判断軸は『別リソースか、同一リソースの絞り込みか』です」

この設問が測っているのは **「A/B の知識」ではなく「前提を疑い、条件で場合分けして語れるか」**。

:::message
**急いでいる人はここで読み終えて大丈夫。** 「パス＝別リソース／クエリ＝絞り込み」。この一行さえ持ち帰れば、実務の URL 設計はほぼ迷わなくなる。
:::

![](/images/20260628-rest-api-design_fig1.png)

上の図は、URL を設計するときに「パスで分けるか・クエリで分けるか」の判断を、たった 2 つの問いで決められるよう整理したものです。「型・権限・ライフサイクルが違う？」と「ID で個別指定する？」の Yes / No だけで答えが出ます。

## 第1部：そもそも API とは何か

### API = システム同士の「契約」

API（Application Programming Interface）は、直訳すると「アプリケーション同士をつなぐ接点」だが、実務的には **「呼び出す側と呼び出される側のあいだの契約（contract）」** と捉えるのが一番しっくりくる。

レストランの比喩で考えると腑に落ちやすい。

- あなた（クライアント）はメニュー（API 仕様）を見て注文する
- キッチン（サーバー）が中で何をしているかは知らなくていい
- 「この注文をすれば、この料理が出てくる」という約束だけが共有されている

ここで重要なのは、**あなたはキッチンの内部を一切知らなくていい**という点だ。コンロがガスか IH か、シェフが何人いるか、冷蔵庫のメーカーは何か。そんなことはどうでもいい。メニューに「カルボナーラ ¥1,200」と書いてあり、注文すればそれが出てくる、という約束だけが意味を持つ。

この「中身を知らなくても、決められた入力に対して決められた出力が返る」という性質を **抽象化（abstraction）** と呼ぶ。API の本質はこの抽象化にある。そしてこの抽象化が、ソフトウェア開発に 2 つの巨大な恩恵をもたらす。

- **呼び出す側**は、相手の実装の詳細から切り離される（中身が変わっても、約束さえ守られれば動き続ける）
- **提供する側**は、約束を守る限り、内部を自由に作り替えられる（DB を MySQL から PostgreSQL に変えても、API の形が同じなら利用者に影響しない）

つまり API とは、**「変わってもいい部分」と「変えてはいけない部分」のあいだに引く境界線**だ。この境界線をどこに引くか、どんな形にするか。それが「API 設計」という営みのすべてだと言ってもいい。

### API にはいくつもの種類がある

「API」という言葉は広い。今日 Web エンジニアが「API」と言うときはほぼ Web API を指すが、本来はもっと広い概念だ。整理しておこう。

| 種類 | 例 | 通信相手 | 失敗するか |
| --- | --- | --- | --- |
| ライブラリ API | `Array.prototype.map()` | 同一プロセス内（メモリ上） | ほぼしない |
| OS API（システムコール） | `open()`, `read()` | カーネル | ときどき |
| Web API | `GET /users` | ネットワーク越しのサーバー | **頻繁に** |

この記事の主題は **Web API**、つまりネットワーク越しに HTTP で叩く API だ。

ここで強調しておきたいのは、表の一番右の列だ。**Web API は、ネットワークを挟むことで「失敗」が日常になる**。ライブラリ関数を呼ぶときに「相手のサーバーがダウンしているかも」「途中で通信が切れるかも」「レスポンスが返ってくるのが 3 秒後かも」とは考えない。しかし Web API ではそれが当たり前に起きる。

この一点、**遅延がある・失敗する・相手が見えない**という事実こそが、Web API 設計のあらゆる難しさの源泉になる。だからこそ「契約」をきっちり決め、失敗時の振る舞いまで含めて設計する必要がある。あとで出てくる冪等性・ステータスコード・リトライといった概念は、すべてこの「ネットワークは信頼できない」という前提から生まれている。

## 第2部：Web API の土台。HTTP の基礎

REST を理解するには、その土台である HTTP を避けて通れない。REST は HTTP の上に成り立つスタイルだからだ。ここを飛ばすと、後半の設計の話が「意味のわからないお作法の暗記」になってしまう。逆にここを丁寧に押さえると、REST の設計判断がすべて「HTTP の素直な使い方」として腑に落ちる。

### リクエストとレスポンスの構造

HTTP 通信は、クライアントが**リクエスト**を送り、サーバーが**レスポンス**を返す、というシンプルな往復で成り立つ。まずこの「往復が 1 回で完結する」という点を押さえておく。これがのちのステートレス制約の伏線になる。

リクエストの中身を分解してみよう。

```http
GET /api/v1/users/42 HTTP/1.1
Host: example.com
Authorization: Bearer eyJhbGc...
Accept: application/json
```

- **メソッド + パス + バージョン**（`GET /api/v1/users/42 HTTP/1.1`）: 「何を」「どうしたいか」と、使う HTTP のバージョン
- **ヘッダー**（`Host`, `Authorization`, `Accept`）: 付帯情報。誰が、何の形式を望み、どこ宛てか
- **ボディ**（GET にはないが、POST などにはある）: 送るデータ本体

レスポンスも同じ構造だ。

```http
HTTP/1.1 200 OK
Content-Type: application/json
Cache-Control: max-age=60

{ "id": 42, "name": "Ishizaka" }
```

- **ステータスコード**（`200 OK`）: 結果がどうだったか
- **ヘッダー**（`Content-Type`, `Cache-Control`）: 中身の形式やキャッシュの指示など
- **ボディ**: 返ってきたデータ本体

**リクエストとレスポンスが、ほぼ同じ「メソッド/ステータス + ヘッダー + ボディ」という構造をしている**ことに注目してほしい。HTTP はこの単純な型の繰り返しでできている。

![](/images/20260628-rest-api-design_fig2.png)

上の図のとおり、HTTP の通信は「行 1 = 動詞 / 結果」「ヘッダー = 付帯情報」「ボディ = データ本体」の 3 段でできていて、リクエストとレスポンスはほぼ対称です。この対称性が分かると、以降の話がすべて「この 3 段のどこに何を入れるか」の話に見えてきます。

### HTTP メソッド。「動詞」を表現する

REST 設計で最も重要な要素の一つが HTTP メソッドだ。これは **リソースに対して何をしたいか（動詞）** を表す。主要なものを網羅しておく。

| メソッド | 意味 | 安全 | 冪等 | ボディ |
| --- | --- | --- | --- | --- |
| `GET` | 取得する | ✅ | ✅ | なし |
| `HEAD` | ヘッダーだけ取得（本体なし） | ✅ | ✅ | なし |
| `POST` | 新規作成する／処理を実行する | ❌ | ❌ | あり |
| `PUT` | 全体を置き換える | ❌ | ✅ | あり |
| `PATCH` | 一部を更新する | ❌ | △ | あり |
| `DELETE` | 削除する | ❌ | ✅ | 任意 |
| `OPTIONS` | 利用可能なメソッドを問い合わせる | ✅ | ✅ | なし |

ここに出てくる **「安全性」と「冪等性」** という 2 つの概念は、地味だが実務で猛烈に効いてくる。腰を据えて理解しておきたい。

#### 安全性（Safe）

**そのメソッドがサーバーの状態を変更しないこと**を「安全」と言う。`GET` は何度叩いてもデータが変わらない。だから次のようなことが成り立つ。

- 検索エンジンのクローラーは、リンクを見つけたら安心して `GET` を投げられる
- ブラウザは、ユーザーがリンクにマウスを乗せた瞬間に勝手にプリフェッチできる
- CDN は気軽にキャッシュできる

これらが成立するのは「`GET` は何も壊さない」という保証があるからだ。逆に言えば、**`GET` で状態を変える設計をした瞬間、これらの前提がすべて崩壊する**。

```
# 絶対にやってはいけない例
GET /users/42/delete   ← GETなのに削除している
```

こういう URL を作ると、クローラーがリンクをたどっただけでユーザーが消える、という悪夢が起きる。実際、過去に「管理画面のリンクをプリフェッチ機能付きブラウザで開いたら、クローラーが全データを削除した」という事故が報告されている。**「`GET` は安全」は単なるお行儀ではなく、エコシステム全体が依存している約束**だと理解しておこう。

#### 冪等性（Idempotent）

**同じリクエストを何回送っても、サーバーの最終状態が変わらないこと**を「冪等」と言う。少し抽象的なので具体例で。

- `DELETE /users/42` を 1 回送る → 42 が消える
- `DELETE /users/42` をもう 2 回送る → すでに消えているので、最終状態は「42 が消えた状態」のまま

何回送っても結果（最終状態）は同じ。だから**ネットワークが不安定でリトライしても安全**だ。「レスポンスが返ってこなかったから、念のためもう一度送る」を躊躇なくできる。

一方 `POST /users` は冪等ではない。

- `POST /users`（name: 石坂）を送る → ユーザー A が作られる
- レスポンスが返ってこないのでリトライ → ユーザー B が作られてしまう（**二重登録**）

ここが分散システムの落とし穴だ。ネットワークでは「リクエストは届いたがレスポンスが返ってこなかった」が普通に起きる。このとき冪等なメソッドならリトライで解決するが、`POST` のような非冪等メソッドはリトライが二重作成を生む。

:::message
`POST` を冪等にしたいときの定石が **冪等性キー（Idempotency-Key ヘッダー）** だ。クライアントがリクエストごとにユニークなキーを発行して送り、サーバーは「このキーは処理済み」と覚えておく。同じキーで再送が来たら、新規作成せずに前回の結果を返す。Stripe の決済 API などがこの方式を採用している。実装例は第 5 部で詳しく扱う。
:::

`PATCH` の冪等性が「△」なのは、**書き方による**からだ。`{ "name": "石坂" }` のような「絶対値の上書き」なら冪等だが、`{ "op": "increment", "value": 1 }` のような「相対的な操作」だと、送るたびに値が増えるので冪等でなくなる。

![](/images/20260628-rest-api-design_fig3.png)

上の図は、主要 5 メソッドを「安全か（状態を変えないか）」と「冪等か（何回送っても結果が同じか）」の 2 軸でマッピングしたものです。`GET` は両方 ◯、`DELETE` / `PUT` は冪等のみ ◯、`POST` はどちらも × という位置関係が一目で見えます。リトライ可否やキャッシュ可否は、この 2 軸から自動的に決まります。

### ステータスコード。「結果」を表現する

レスポンスの先頭で、結果を 3 桁の数字で伝える。**先頭の 1 桁でカテゴリが決まる**のがポイントだ。

| レンジ | 意味 | 代表例 |
| --- | --- | --- |
| `1xx` | 情報（処理中） | `100 Continue` |
| `2xx` | 成功 | `200 OK`, `201 Created`, `204 No Content` |
| `3xx` | リダイレクト | `301 Moved Permanently`, `304 Not Modified` |
| `4xx` | **クライアント**側のエラー | `400`, `401`, `403`, `404`, `409`, `422`, `429` |
| `5xx` | **サーバー**側のエラー | `500`, `502`, `503` |

`4xx` と `5xx` の切り分けが特に大事だ。**「悪いのは呼び出した側か、提供した側か」** をここで明示する。これを正しく返すかどうかで、API を叩く側のデバッグ体験がまるで変わる。

よく使うものを一つずつ見ておこう。

**2xx（成功）**

- `200 OK`: 成功。`GET` の取得成功や、`PATCH` の更新成功で返す
- `201 Created`: 作成成功。`POST` でリソースを作ったとき。作ったリソースの URL を `Location` ヘッダーに入れるのが作法
- `204 No Content`: 成功したが返す本体がない。`DELETE` 成功や、本体不要な更新で使う

**4xx（クライアントのミス）**

- `400 Bad Request`: リクエストの形式自体が壊れている（JSON が壊れている等）
- `401 Unauthorized`: **認証**されていない（誰だかわからない）
- `403 Forbidden`: 認証は済んでいるが**権限**がない（誰だかはわかるが、ダメ）
- `404 Not Found`: リソースが存在しない
- `405 Method Not Allowed`: そのリソースにそのメソッドは使えない（`/users` に `DELETE` 等）
- `409 Conflict`: 競合（すでに同じメールで登録済み、楽観ロックの衝突 等）
- `422 Unprocessable Entity`: 形式は正しいが内容のバリデーションに失敗（email の形式が不正 等）
- `429 Too Many Requests`: レート制限超過

**5xx（サーバーのミス）**

- `500 Internal Server Error`: サーバー側で想定外の例外。「とりあえず 500」は避け、ハンドリングできるものは適切な 4xx に倒す
- `502 Bad Gateway`: ゲートウェイ／プロキシが上流から不正な応答を受けた
- `503 Service Unavailable`: 一時的に処理不能（メンテナンス、過負荷）。`Retry-After` ヘッダーで再試行の目安を返せる

:::message alert
現場で頻発する 3 つの混同に注意。
- **`401` と `403`**: 「認証(401)＝あなたが誰か不明」「認可(403)＝あなたは分かるが権限なし」。別物。
- **`400` と `422`**: 「`400`＝構文が壊れている」「`422`＝構文は正しいが意味的にダメ（バリデーションエラー）」。
- **`404` の使い分け**: 存在しないから 404 なのか、権限がなくて見せたくないから（あえて）404 なのか。セキュリティ上、存在を隠すために 403 ではなく 404 を返す設計もある。
:::

![](/images/20260628-rest-api-design_fig4.png)

上の図は、よく使うステータスコードを 5 レンジのカードに並べた早見表です。`2xx` 成功・`3xx` リダイレクト・`4xx` クライアント側のミス・`5xx` サーバー側のミス、と「責任の所在」で見ると意味の取り違えが減ります。

### ヘッダー。通信のメタ情報

ボディが「中身」だとすれば、ヘッダーは「中身に関する付帯情報」だ。重要なものを押さえておく。これらは後の章のキャッシュ・認証・コンテンツネゴシエーションで全部出てくる。

| ヘッダー | 向き | 役割 |
| --- | --- | --- |
| `Content-Type` | 両方 | ボディの形式（`application/json` 等） |
| `Accept` | リクエスト | クライアントが望む形式 |
| `Authorization` | リクエスト | 認証情報（`Bearer <token>` 等） |
| `Cache-Control` | 両方 | キャッシュの可否・有効期間 |
| `ETag` | レスポンス | リソースのバージョン識別子 |
| `Location` | レスポンス | 作成された／移動先のリソース URL |
| `Retry-After` | レスポンス | 再試行までの推奨待機時間 |

**コンテンツネゴシエーション**という仕組みも知っておくと一段深い。クライアントが `Accept: application/json` と送り、サーバーが「では JSON で返します」と応じて `Content-Type: application/json` で返す。この「形式のすり合わせ」のことだ。同じ URL でも `Accept: application/xml` なら XML を返す、といった出し分けができる。

### HTTP のバージョンにも軽く触れておく

REST の設計そのものには直接関係しないが、土台として知っておくと視野が広がる。

- **HTTP/1.1**: 1 接続で 1 リクエストずつ（実際にはキープアライブで使い回すが、基本は直列）。長く Web を支えてきた
- **HTTP/2**: 1 接続で複数リクエストを並行処理（多重化）。ヘッダー圧縮もある。同一サーバーへの多数のリクエストが速くなった
- **HTTP/3**: トランスポートが TCP から QUIC（UDP ベース）に。接続確立が速く、パケットロスに強い

REST API を設計する側は普段ここを意識しないが、「なぜ昔は画像をスプライトにまとめてリクエスト数を減らしたのに、今はやらなくていいのか（HTTP/2 で多重化されたから）」のような話の背景になる。

## 第3部：REST とは何か

ここでようやく本題の REST に入る。多くの人が「REST = JSON を返す HTTP API」くらいの理解で止まっているが、REST はもっと明確に定義された **アーキテクチャスタイル** だ。

### REST の出自。なぜ生まれたか

REST（REpresentational State Transfer）は、2000 年に Roy Fielding が博士論文で提唱した。彼は HTTP/1.1 の仕様策定にも関わった人物だ。重要なのは、REST が**机上の理想として作られたのではなく、「すでに大成功していた Web」を後追いで理論化したもの**だということ。

Fielding が問うたのは「**Web はなぜ、これほど巨大にスケールし、これほど長く壊れずに進化し続けられたのか**」だった。世界中の無数のサーバーとブラウザが、中央の調整なしに繋がり、しかも HTTP のバージョンが上がっても古いページが動き続ける。この驚異的な性質を生んでいる構造を抽出したものが REST だ。

だから REST は**特定の技術や規格ではなく、「こういう制約を守ると Web のようにスケールするよ」という設計原則の集合**になっている。ここを誤解していると設計判断がブレるので、何度でも強調しておく。

### REST の6つの制約

Fielding は REST を 6 つの制約（constraints）で定義した。一つずつ、なぜそれがスケーラビリティに効くのかとセットで見ていく。

**1. クライアント・サーバー分離**
UI（クライアント）とデータストレージ（サーバー）の関心を分離する。片方の都合でもう片方を止めずに、それぞれを独立して進化させられる。Web で言えば「ブラウザを更新しなくてもサーバー側を改修できる」「サーバーを変えなくてもブラウザを乗り換えられる」状態。

**2. ステートレス（最重要）**
サーバーはリクエスト間でクライアントの状態を保持しない。**各リクエストは、それ単体で処理に必要な情報をすべて含む**。

これがどういうことか、対比で見ると鮮明だ。

- **ステートフル**（悪い）: ログイン状態をサーバーのメモリに持つ。すると「この人を前に処理したサーバー」でしか続きを処理できない。サーバーを増やしても、特定の人を特定のサーバーに固定（スティッキーセッション）する必要が出てくる
- **ステートレス**（良い）: 毎回トークンを送り、サーバーは記憶を持たない。すると**どのサーバーがどのリクエストを処理してもいい**。ロードバランサーが適当に振り分けられる

つまりステートレスは、**サーバーを横に並べるだけでスケールできる（水平スケール）** ための前提だ。REST が「スケーラブル」と言われる最大の理由がここにある。冒頭の面接問題の「スケーラブル」というキーワードも、根っこはこの思想に繋がっている。

![](/images/20260628-rest-api-design_fig6.png)

上の図は、ステートフル（左）とステートレス（右）でリクエストがサーバーに振り分けられる様子を対比したものです。左はセッションを覚えているサーバーに固定で送らねばならず、右はどのサーバーでも処理できるため自由に並列化できます。「スケーラブル」の正体はここの構造差です。

**3. キャッシュ可能**
レスポンスは「キャッシュしていいか／どれくらい有効か」を明示する。`GET` が安全（状態を変えない）だからこそ、ブラウザや CDN が安心して結果を取っておける。これにより、同じリクエストでサーバーまで往復しなくて済み、負荷が劇的に下がる。`Cache-Control`、`ETag`、`304 Not Modified` がこの制約の実装手段（第 5 部で詳述）。

**4. 統一インターフェース（REST の核心）**
リソースを URI で識別し、HTTP メソッドという共通の動詞で操作する。これにより、**どの API も同じ「叩き方の文法」になる**。`GET /users/42` の意味は、ドメインが違っても同じ。この一貫性が、ツール・キャッシュ・プロキシといった汎用部品を成立させる。次節でさらに掘り下げる。

**5. 階層化システム**
クライアントは、直接サーバーと話しているのか、間にロードバランサー・CDN・プロキシ・APIゲートウェイが挟まっているのかを意識しなくていい。**中間層を自由に挿入・除去できる**。だからキャッシュ層を後から足したり、認証をゲートウェイに切り出したりできる。

**6. コードオンデマンド（任意）**
サーバーがクライアントに実行可能コードを送れる（ブラウザに JavaScript を送るのがまさにこれ）。6 つの中でこれだけは必須ではない。

:::message
これら 6 つをすべて満たすものを Fielding は厳密な意味での REST と呼ぶ。だが現実の「REST API」の多くは、次に説明する HATEOAS を満たさない **"RESTish"（REST 風）** だ。これは悪いことではない。原理主義に陥らず「自分のは Level いくつの REST か」を自覚できることのほうが、実務では価値がある。
:::

![](/images/20260628-rest-api-design_fig5.png)

上の図は、REST を成立させる 6 つの制約をひとつの俯瞰図にまとめたものです。中心の「統一インターフェース」を、ステートレス・キャッシュ可能・クライアント／サーバー分離・階層化システム・コードオンデマンドの 5 つが取り囲む構造で見ると、それぞれが何のためにある制約なのかが整理されます。

### リソース指向。「動詞でなく名詞」という発想の転換

統一インターフェースを実装に落とすときの肝は、**「動詞でなく名詞（リソース）で考える」** ことだ。これは慣れるまで違和感があるが、腑に落ちると設計が一気に楽になる。

RPC 的な発想だと、API は「やりたい操作（動詞）」の一覧になる。

```
POST /getUser
POST /createUser
POST /deleteUser
POST /updateUserName
POST /banUser
```

操作が増えるたびにエンドポイントが増え、命名にも一貫性がなくなっていく。`getUser` か `fetchUser` か `retrieveUser` か。チームで延々と揉める。

REST 的な発想だと、API は「リソース（名詞）」の一覧になり、操作は HTTP メソッドが担う。

```
GET    /users/42      取得
POST   /users         作成
PATCH  /users/42      更新
DELETE /users/42      削除
```

**「何をするか」をエンドポイント名に書くのをやめ、「何に対して」をパスで、「どうするか」をメソッドで表す。** この役割分担こそが統一インターフェースの正体だ。動詞は HTTP がすでに用意してくれている（GET/POST/PATCH/DELETE）ので、自分で命名する必要がない。だから一貫性が保たれる。

### CRUD に収まらない「アクション」をどう表現するか

リソース指向に慣れてくると、必ずぶつかる壁がある。「**ユーザーを ban する**」「**カートを精算する**」のような、CRUD（作成・取得・更新・削除）に素直に収まらない操作だ。これらをどう REST で表現するか。代表的な 3 つのアプローチがある。

```
# 1. 状態をリソースとして PATCH で更新する（最も REST らしい）
PATCH /users/42        { "status": "banned" }

# 2. サブリソースを「作成」する（イベントをリソースと捉える）
POST /users/42/bans
POST /carts/42/checkout

# 3. 動詞をパスに含める（純粋さは劣るが、明快で現実的）
POST /users/42/ban
```

どれも一長一短で、宗教論争になりがちな領域だ。**「ban という行為そのものを `ban` というリソース（記録）として作成する」** と捉え直すと 2 番の形に落とせて REST 的な一貫性を保てる、というのが一つの定番解。一方、可読性を優先して 3 番を割り切って使うチームも多い。**大事なのは「正解は一つ」と思い込まないこと**。この感覚は、まさに冒頭の面接問題と同じ構図だ。

### Richardson 成熟度モデル。自分の REST 度を測る

「自分の API はどれくらい REST なのか」を測る有名なものさしが、**Richardson Maturity Model（RMM）** だ。Level 0 から 3 までの 4 段階で、上に行くほど REST の理念に忠実になる。

**Level 0 、 HTTP をトンネルとして使うだけ**
1 つのエンドポイントに全部 `POST` で投げ、ボディの中身で操作を指定する。HTTP の意味をほとんど使っていない。

```
POST /api    { "action": "getUser", "id": 42 }
POST /api    { "action": "deleteUser", "id": 42 }
```

**Level 1 、 リソースの導入**
URL でリソースを区別するようになる。ただしメソッドはまだ `POST` 一辺倒なことが多い。

```
POST /users/42   { "action": "get" }
POST /users/42   { "action": "delete" }
```

**Level 2 、 HTTP メソッドの活用（実務の「REST API」の大半はここ）**
リソース（名詞）＋ HTTP メソッド（動詞）＋ 適切なステータスコードを使う。冒頭の A 案・B 案もこのレベルの話をしている。

```
GET    /users/42   → 200
DELETE /users/42   → 204
```

**Level 3 、 HATEOAS（ハイパーメディア）**
レスポンスの中に「次に取れる行動へのリンク」を含める。クライアントは URL を組み立てる知識を持たなくても、返ってきたリンクをたどって操作できる。

```json
{
  "id": 42,
  "name": "Ishizaka",
  "status": "active",
  "_links": {
    "self":      { "href": "/users/42" },
    "orders":    { "href": "/users/42/orders" },
    "deactivate":{ "href": "/users/42", "method": "DELETE" }
  }
}
```

Web ページがリンクをたどって遷移できるのと同じように、API もレスポンス内のリンクをたどって操作できる、という発想だ。理論的には美しく、クライアントとサーバーの結合を最小にできる。

ただし HATEOAS は実装・運用コストが高く、採用しているサービスは多くない。**多くの実務 API は Level 2 で止まっている**し、それで困っていない。だが「自分のは Level 2 で、Level 3 にはこういう理想がある」と自覚できると、設計の議論で一段深い視座が持てる。面接でもここを語れると強い。

![](/images/20260628-rest-api-design_fig7.png)

上の図は、API の「REST 度」を 4 段階の階段で示したものです。Level 0 が HTTP トンネル、Level 1 でリソース導入、Level 2 で HTTP メソッドを活用、Level 3 で HATEOAS。実務の大半が Level 2 にいるという現実と、その上にもう一段あることを同時に見せます。

## 第4部：URL 設計の原則。そして面接問題の回収

ここまでの基礎を踏まえて、いよいよ冒頭の面接問題に戻る。鍵になるのは **「パスとクエリ、それぞれの役割は何か」** という、たった一つの問いだ。

### パスとクエリは役割が違う

| | 役割 | 例 |
| --- | --- | --- |
| **パス**（path） | **別のリソース**を指し示す | `/users/42/orders` |
| **クエリ**（query） | **同一リソース**の絞り込み・並び替え・ページング | `/users?status=active&sort=created_at` |

これが原則のすべてだ。言い換えるとこうなる。

- **パスで分けるべきもの** = 別の実体。レスポンスの型が違う、権限が違う、ライフサイクルが違う
- **クエリで分けるべきもの** = 同じ型のコレクションに対する、条件付けや見せ方の調整

もう一つの覚え方として、**「パスは "What"（何を）、クエリは "Which/How"（どれを・どう見せるか）」** と捉えてもいい。`/users/42` で「何を」を確定させ、`?fields=name` で「それをどう見せるか」を指定する、という分担だ。

### よくある URL 設計のパターン集

原則を具体に落とすために、典型的な形を一通り並べておく。

```
# コレクションと個別リソース
GET    /users           ユーザー一覧（コレクション）
GET    /users/42        ID=42 のユーザー（個別）
POST   /users           ユーザーを作成

# ネストしたリソース（所属関係を表す）
GET    /users/42/orders        ユーザー42の注文一覧
GET    /users/42/orders/7      ユーザー42の注文7

# シングルトンリソース（1つしか存在しないもの）
GET    /users/42/profile       ユーザー42のプロフィール（1人に1つ）
GET    /me                     ログイン中の自分

# 絞り込み・並び替え・ページング（すべてクエリ）
GET    /users?role=admin&status=active
GET    /users?sort=-created_at
GET    /users?page=2&limit=20
```

ネストは便利だが、**深くしすぎない**のがコツ。`/users/42/orders/7/items/3/reviews` のように 3 階層以上ネストすると、URL が硬直化し、`item` 単体を取りたいだけなのに親をすべて辿る羽目になる。深い関係はネストせず、`/order-items/3` のようにトップレベルのリソースとして用意し、関連は ID で繋ぐほうが柔軟なことが多い。

### 面接問題の回収

ここまでの原則を武器に、A 案と B 案を解剖する。

```
# A案: パスで分ける
GET /api/v1/users/profile
GET /api/v1/users/settings

# B案: クエリで分ける
GET /api/v1/users?type=profile
GET /api/v1/users?type=orders
```

**問うべきは「`profile` と `orders` と `settings` は、別リソースか、それとも同一リソースの絞り込みか」** だ。そして答えは前提によって割れる。両方のケースを丁寧に見る。

#### ケース1：これらが別リソースの場合 → A（パス）が正しい

`profile`（プロフィール）と `orders`（注文履歴）と `settings`（設定）が、それぞれ別のスキーマ・別の権限・別のページネーションを持つ"別物"だとする。このとき B のように `?type=` で 1 エンドポイントに束ねると、**返る型が `type` の値によって変わる多態（polymorphic）エンドポイント**になり、以下の問題が芋づる式に発生する。

- **型定義・ドキュメントが破綻する**: 1 エンドポイントの戻り値が `type` 次第でバラバラ。OpenAPI で表現しづらく、クライアントの型も `Profile | Order | Settings` のユニオンになって扱いにくい
- **横断的関心事をリソース単位で分けられない**: 「`orders` だけ厳しいレート制限」「`settings` だけ管理者のみ」のような、認可・レート制限・キャッシュのリソース別ポリシーが書けない。全部 `/users` という 1 つの口になってしまうから
- **ハンドラが神メソッド化する**: サーバー側は `switch (type)` の巨大な分岐に育ち、`type` が増えるたびに 1 つの関数が肥大する
- **サービス分割ができない**: マイクロサービス化で「`orders` だけ別チーム・別サービスに切り出す」をやろうとしても、パスが同じなのでルーティングで分離できない

つまりこのケースでは、**B は短期的に楽に見えて、スケールするほど苦しくなる**。「スケーラブル」を問う設問の意図に照らすと、A が正解になる。

#### ケース2：これらが同一スキーマの汎用ドキュメントの場合 → B（クエリ）が正しい

一方、`profile` / `orders` / `settings` が実は**同じスキーマを持つ汎用的なドキュメント**で、`type` はその出し分けの条件でしかない、という設計だとする。この場合は逆転して、B が正しい。

- **拡張に強い（Open/Closed 原則）**: 新しい `type` の追加が、ルート追加＆デプロイ（A）ではなく、許可値を 1 個増やすだけで済む
- **ロジックを集約できる**: 同一スキーマなのだから、取得・認可・バリデーションを 1 箇所にまとめられる。A だと同じようなハンドラが type の数だけ並ぶ
- **他のクエリと自然に合成できる**: `?type=profile&fields=...&page=...&sort=...` のように、絞り込み・整形のクエリと素直に組み合わさる。これはクエリパラメータの本来の使い方そのもの

### 本当の正解は「場合分けして語ること」

つまりこの面接問題の満点回答は **「A」でも「B」でもない**。こうだ。

> これらが**異なるリソース**（別スキーマ・別権限・別ライフサイクル）なら、別リソースなのでパスで分ける **A** が正しい。一方、これらが**同一スキーマの可変ドキュメント**で `type` が単なる出し分けなら、クエリで合成できる **B** が正しい。判断基準は一貫していて、「**別リソースか、同一リソースの絞り込みか**」だ。

これが、出題者の意図、おそらく「短絡的に B を選ぶ人をふるい落とす罠」を完全に見抜いた回答になる。**API 設計で問われているのは「答えの暗記」ではなく「判断基準を持っているか」** という、この記事を貫くテーマがここに凝縮されている。

![](/images/20260628-rest-api-design_fig8.png)

上の図は、冒頭の A 案・B 案を「前提」で分岐させて見直す全体像です。別リソースなら A、同一スキーマの絞り込みなら B、と二股に分かれていく構造を 1 枚にすると、「即答した時点で不正解」が腑に落ちます。

:::message
ちなみに A 案も完璧ではない。特定ユーザーを指すなら本来 `GET /users/{id}/profile` のように **ID を含めるべき**で、`/users/profile` だと「誰の profile か」が曖昧（暗黙に「ログイン中の自分」を指すなら `/me/profile` のほうが明示的で安全）。ここまで指摘できると盤石だ。
:::

### B（クエリ）を採用するときの実務的ガード

ケース 2 のように B が妥当でも、雑に実装すると事故る。最低限これは守る。

- **許可値を enum でホワイトリスト化**する。`type` に任意文字列を受け付けない（`?type=__proto__` のような攻撃やタイポを弾く）
- `type` ごとに戻り値の型が変わるなら、それは結局多態だ。スキーマを揃えるか、**discriminated union**（`{ "kind": "profile", ... }` のように種別フィールドを明示）にして、クライアントが安全に分岐できるようにする
- 必須の `type` が欠けていたら、黙って全件返したりせず **`400 Bad Request`** を返す

## 第5部：実務で必要になる設計パターン

原則がわかっても、実際の API には「現実の都合」がある。ここでは現場で必ず必要になる定番パターンを、一つずつ丁寧に扱う。

### バージョニング

API は一度公開すると簡単には壊せない。既存のクライアントが動かなくなるからだ。だから破壊的変更に備えてバージョンを切る。主な方式は 3 つ。

```
# 1. パスに含める（最も一般的・推奨）
GET /api/v1/users

# 2. ヘッダーで指定する（URL がきれいだがキャッシュ・確認がやや面倒）
GET /api/users
Accept: application/vnd.myapp.v1+json

# 3. クエリで指定する（手軽だが推奨度は低い）
GET /api/users?version=1
```

実務では **パス方式（`/v1/`）が圧倒的に多い**。理由は、URL を見ただけでバージョンがわかり、ルーティングもキャッシュも素直になり、ブラウザのアドレスバーで直接試せるから。冒頭の問題の `/api/v1/` もこれだ。

補足として、**「いつバージョンを上げるか」** の基準も持っておきたい。フィールドを「追加する」だけなら既存クライアントは壊れない（無視するだけ）ので、バージョンは上げないのが普通。**フィールドの削除・リネーム・意味変更・必須化**といった破壊的変更のときだけ `v2` を切る。むやみに上げると古いバージョンの保守地獄になる。

### ページネーション

全件返すわけにはいかないので、データを分割して返す。代表的な 2 方式を、トレードオフまで含めて見る。

**オフセット方式**

```
GET /users?limit=20&offset=40    # 41件目から20件
```

「○件目から△件」と指定する直感的な方式。SQL の `LIMIT/OFFSET` にそのまま対応する。だが 2 つの弱点がある。

- **深いページが遅い**: `offset=1000000` だと、DB は 100 万件読み飛ばす必要があり重い
- **ズレる**: ページをめくっている最中にデータが挿入・削除されると、項目の重複や抜けが起きる

**カーソル方式**

```
GET /users?limit=20&cursor=eyJpZCI6MTIzfQ   # このカーソル以降を20件
```

「最後に見た位置（カーソル）の続きから」と指定する方式。カーソルは通常「最後の項目の ID や作成日時」をエンコードしたもの。**深いページでも速く、データが増減してもズレにくい**。代償として「7 ページ目に飛ぶ」のような任意ページへのジャンプはできない。

**SNS のタイムラインや無限スクロール、大規模データではカーソル方式**が選ばれる。管理画面の「全 N ページ・任意ジャンプ」が要るならオフセット方式、と使い分ける。

レスポンスには「次があるか」を含めると親切だ。

```json
{
  "data": [ /* ... */ ],
  "pagination": {
    "next_cursor": "eyJpZCI6MTQzfQ",
    "has_more": true
  }
}
```

![](/images/20260628-rest-api-design_fig9.png)

上の図は、ページネーションの 2 方式を「深いページの速度」「データ追加時のズレ」「任意ページへのジャンプ」の 3 観点で比較したものです。カーソル方式は速くてズレに強い代わりに任意ジャンプができない、というトレードオフが一目で確認できます。

### フィルタリング・ソート・部分取得

これらはすべて「同一リソースの絞り込み・見せ方の調整」なので、**クエリパラメータで表現する**のが原則だ（第 4 部そのもの）。慣習的な書き方を挙げておく。

```
GET /users?status=active&role=admin   # 複数条件（AND）
GET /users?sort=-created_at,name      # 降順は - を前置、複数キーはカンマ区切り
GET /users?fields=id,name             # 必要なフィールドだけ取得（オーバーフェッチ低減）
GET /users?created_after=2026-01-01   # 範囲指定
GET /users?q=ishizaka                 # 全文検索
```

これらに厳密な標準はなく、API ごとに方言がある。だからこそ**自分の API 内で一貫した規約を決める**ことが大事だ（`sort` の降順表現を `-name` にするのか `name:desc` にするのか、最初に決めて全エンドポイントで揃える）。

### エラーレスポンスの設計

ステータスコードだけでなく、ボディでも構造化されたエラー情報を返すと、利用側のデバッグが圧倒的に楽になる。

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Validation failed for 1 field",
    "details": [
      { "field": "email", "issue": "invalid_format" }
    ]
  }
}
```

ポイントは 2 つ。

- **機械可読な `code` と人間可読な `message` を分ける**。`code`（`VALIDATION_ERROR` 等の定数）はプログラムが分岐に使う。`message` は開発者が読む説明。**`message` の文字列でクライアントが分岐する設計は厳禁**。文言を変えただけでクライアントが壊れる
- **`details` でフィールド単位の理由を返す**。フォームのどの項目がなぜダメだったかを、クライアントがそのまま画面に反映できる

:::message
エラーフォーマットには **RFC 9457（Problem Details for HTTP APIs）** という標準もある。`type` / `title` / `status` / `detail` / `instance` というフィールドを持つ JSON（`Content-Type: application/problem+json`）で、迷ったらこれに寄せると車輪の再発明を避けられる。
:::

### 認証と認可。別物として設計する

第 2 部のステータスコードでも触れたが、改めて。

- **認証（Authentication）** = 「あなたは誰か」を確認する → 失敗は **`401`**
- **認可（Authorization）** = 「その人にこれをする権限があるか」を確認する → 失敗は **`403`**

Web API での認証の代表的な手段を整理する。

| 方式 | 仕組み | 向くケース |
| --- | --- | --- |
| **API キー** | 固定の文字列を送る | サーバー間連携、社内ツール |
| **Bearer トークン（JWT 等）** | 署名付きトークンを `Authorization` に載せる | ユーザー認証全般 |
| **OAuth 2.0** | 第三者にアクセス権を委譲する | 「Google でログイン」等の連携 |

このうち **JWT（JSON Web Token）** はステートレス制約（第 3 部）と相性が良いので少し詳しく。JWT は 3 つの部分をドットで繋いだ文字列だ。

```
ヘッダー.ペイロード.署名
eyJhbGc...  .  eyJzdWIi...  .  SflKxwRJ...
```

- **ヘッダー**: 署名アルゴリズムなど
- **ペイロード**: ユーザー ID や有効期限などの情報（Base64 で誰でも読めるので**秘密を入れてはいけない**）
- **署名**: サーバーの秘密鍵で作る。改ざんを検知できる

ミソは、**サーバーがセッションを覚えなくても、トークン自体に「誰か」が入っており、署名で本物だと検証できる**こと。だからどのサーバーで受けても処理でき、水平スケールできる。これがステートレスの具体的な恩恵だ。

:::message alert
JWT の注意点。ペイロードは暗号化ではなく**ただの Base64**なので中身は丸見え。機密情報は入れない。また、サーバーが状態を持たない裏返しとして、**発行済みトークンの即時無効化（ログアウトや BAN の即時反映）が難しい**。有効期限を短くして、リフレッシュトークンと組み合わせる設計が定番。
:::

### キャッシュ。`ETag` と条件付きリクエスト

REST の「キャッシュ可能」制約（第 3 部）を実装する具体策を見る。鍵は **`ETag`** と **`304 Not Modified`** だ。

1. クライアントが `GET /users/42` する
2. サーバーがデータと一緒に `ETag: "abc123"`（中身のバージョン識別子）を返す
3. 次回、クライアントは `If-None-Match: "abc123"` を付けて問い合わせる
4. 中身が変わっていなければ、サーバーは**本体を送らず `304 Not Modified` だけ返す**

```http
# 2回目のリクエスト
GET /api/v1/users/42 HTTP/1.1
If-None-Match: "abc123"

# 変わっていなければ
HTTP/1.1 304 Not Modified
```

これで、変わっていないデータを何度も転送せずに済む。`Cache-Control: max-age=60` を併用すれば「60 秒間はサーバーに問い合わせず手元のキャッシュを使う」も指定できる。**`GET` が安全だからこそ、この仕組みが安心して使える**。第 2 部・第 3 部の伏線がここで回収される。

### レート制限。`429` と `Retry-After`

API を守るために、一定時間あたりのリクエスト数を制限する。超過したら **`429 Too Many Requests`** を返す。このとき、クライアントが正しく振る舞えるよう、ヘッダーで状況を伝えるのが親切だ。

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 30
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1719500000
```

- `Retry-After`: 何秒後に再試行していいか
- `X-RateLimit-*`: 上限・残り・リセット時刻（慣習的なヘッダー）

クライアント側は `429` を受けたら、`Retry-After` 秒待って**指数バックオフ**（待ち時間を徐々に伸ばす）でリトライする、というのが行儀の良い実装だ。

### 冪等性キー。`POST` を安全にリトライする

第 2 部で予告した、`POST` の二重実行を防ぐ仕組み。流れはこうだ。

1. クライアントがユニークなキー（UUID 等）を生成し、`Idempotency-Key: <uuid>` ヘッダーで送る
2. サーバーは処理前に「このキーは処理済みか」をストレージで確認する
3. 未処理なら実行し、結果をキーに紐づけて保存する
4. **同じキーで再送が来たら、再実行せずに保存済みの結果を返す**

```http
POST /api/v1/payments HTTP/1.1
Idempotency-Key: 9f8b7c6d-1234-...
Content-Type: application/json

{ "amount": 5000, "currency": "JPY" }
```

これで「決済リクエストを送ったがレスポンスが返ってこず、不安だからリトライしたら二重課金」という事故を防げる。決済のような失敗が許されない `POST` では事実上必須のパターンだ。

### CORS。ブラウザからの呼び出しで必ず出会う壁

ブラウザの JavaScript から、別ドメインの API を叩くときに必ず関係してくるのが **CORS（Cross-Origin Resource Sharing）** だ。ブラウザは安全のため、デフォルトでは「自分と異なるオリジン（ドメイン）」への一部のリクエストを制限している。これを許可するには、**サーバー側が** `Access-Control-Allow-Origin` などのヘッダーで「このオリジンからのアクセスを許す」と明示する必要がある。

```http
HTTP/1.1 200 OK
Access-Control-Allow-Origin: https://app.example.com
Access-Control-Allow-Methods: GET, POST, PATCH, DELETE
```

「ローカルから API を叩いたら CORS エラーが出た」は新人が必ず一度はハマる。**CORS はサーバー側の設定で解決するもの**で、フロント側で小細工するものではない、と覚えておくと迷わない。

## 第6部：やりがちなアンチパターン集

原則の裏返しとして、「これをやると後で苦しむ」典型例をまとめておく。レビューのチェックリストにも使える。

- **`GET` で状態を変える**（`GET /users/42/delete`）: 安全性の破壊。クローラーやプリフェッチで事故る
- **動詞をパスに濫用する**（`/getUserList`, `/doUpdate`）: リソース指向の放棄。Level 0〜1 に逆戻り
- **すべて `200` で返し、ボディの `success: false` で失敗を表す**: HTTP の意味を捨てている。エラーは適切な 4xx/5xx で
- **別リソースをクエリで多態化する**（まさに B 案ケース 1）: 型崩壊・ポリシー分離不能
- **ページネーションがない一覧 API**: データ増加で必ず破綻する。最初から入れる
- **エラーメッセージの文言でクライアントが分岐**: 文言変更で壊れる。機械可読な `code` を返す
- **破壊的変更をバージョンなしで入れる**: 既存クライアントが沈黙して壊れる
- **深すぎるネスト**（`/a/1/b/2/c/3/d/4`）: 硬直化。関連は ID で繋ぐ
- **複数形と単数形が混在**（`/user/42` と `/orders`）: 一貫性がない。**コレクションは複数形で統一**するのが無難
- **JWT のペイロードに秘密情報を入れる**: Base64 で丸見え。事故る

## 第7部：REST の限界とその先

REST は万能ではない。網羅性のために「REST が苦手なこと」と「代替手段」にも触れておく。**新しいものが常に正しいわけではない**。という視点を持つのがこの章のゴールだ。

### REST が苦しくなる場面

**オーバーフェッチ / アンダーフェッチ**
`GET /users/42` は固定の形を返す。

- 「名前だけ欲しい」のに全部返ってくる → **オーバーフェッチ**（無駄な転送）
- 「ユーザーとその注文とその商品」を一度に欲しいのに、`/users/42`→`/users/42/orders`→各 `/products/x` と何度も叩く羽目になる → **アンダーフェッチ / N+1 問題**

`?fields=` や `?include=` である程度緩和できるが、画面ごとに最適な形が違うと限界がくる。

**エンドポイントの爆発**
画面ごとに最適なデータ形が違うと、`/users/42/dashboard-summary` のような画面専用エンドポイントが増殖しがちになる。

### 代替アーキテクチャ

| 方式 | 強み | 弱み | 向くケース |
| --- | --- | --- | --- |
| **GraphQL** | クライアントが必要な形を指定。オーバー/アンダーフェッチ解消 | HTTP キャッシュが効きにくい、複雑性 | 多様なクライアント、複雑な画面 |
| **gRPC** | バイナリで高速、型が厳格、双方向ストリーム | ブラウザから直接叩きにくい、可読性低 | マイクロサービス間、低レイテンシ要求 |
| **tRPC** | TS で型を端から端まで共有、コード生成不要 | TS エコシステムに閉じる | TS で完結するフルスタック |

ただし誤解してはいけないのは、**これらは「REST の上位互換」ではない**ことだ。

- GraphQL は柔軟だが、URL ベースの HTTP キャッシュ（`ETag` や CDN）が効きにくくなる。すべてが `POST /graphql` に集約されるため、第 5 部で見たキャッシュの恩恵を失う
- gRPC は速いが、ブラウザから直接は叩けず（gRPC-Web というラッパが要る）、`curl` で気軽に試すこともできない

REST のシンプルさ・HTTP インフラとの親和性・キャッシュのしやすさ・ツールの豊富さは、依然として強力な武器だ。**「適材適所」であって、新しいものが常に正解ではない。** どの方式を選ぶかは、クライアントの多様性・パフォーマンス要件・チームの習熟度を天秤にかけて判断する。この判断の構図は、結局のところ冒頭の面接問題（前提次第で正解が変わる）とまったく同じだ。

![](/images/20260628-rest-api-design_fig10.png)

上の図は、4 つの方式を「キャッシュのしやすさ」「型の厳格さ」「学習コスト」「向くクライアント多様性」などの軸で並べたものです。「新しい＝強い」ではなく、軸ごとに得意分野が違うことが見えます。

## まとめ

1 問の面接問題から始めて、API の基礎・HTTP・REST・URL 設計・実務パターン・アンチパターン・その先まで、省略せず降りてきた。最後に、幹となる判断基準だけを抜き出しておく。

- API の本質は **「契約」と「抽象化」**。中身を知らずに、決めた入出力だけでつながる。**境界線をどこに引くか**が設計のすべて
- HTTP の **安全性・冪等性** が、キャッシュ・リトライ・スケーラビリティの土台。ここを理解すると REST の作法が「素直な HTTP の使い方」として腑に落ちる
- REST は技術ではなく **制約の集合**。とくに **ステートレス** が水平スケールを可能にする。`GET` が安全だからキャッシュが効く
- リソース指向 ＝ **「動詞でなく名詞」**。何をするかはメソッドが担う。CRUD に収まらない操作は「行為をリソース化」して表現できる
- 自分の REST 度は **Richardson 成熟度モデル**で測れる。多くの実務 API は Level 2
- URL 設計の原則は **「パスは別リソース、クエリは同一リソースの絞り込み」** の一行に集約される
- そして最重要 。 **API 設計は答えの暗記ではなく、判断基準を持っているか**。「A か B か」ではなく「どういう前提ならどちらが正解か」を語れることがゴール

冒頭の面接問題に即答で「B です」と答えていた自分は、半分正解で半分不正解だった。**前提を場合分けして語れて初めて満点になる**。この「前提を疑い、条件で場合分けして語る」感覚こそ、この長い記事を通して一番持ち帰ってほしいものだ。それは API 設計だけでなく、技術選定のあらゆる場面で効いてくる。

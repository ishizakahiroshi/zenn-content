![](/images/20260702-declined-obsidian-for-now_hero.png)

Obsidian の vault を立ち上げてみました。ディレクトリを 4 つ切り、Google Drive で同期を通し、junction で配線して、8 章の plan md まで書きました。骨格は、きれいに整いました。

そして、実運用に踏み出す前に手を止めました。

この記事は「Obsidian が悪い」でも「PKM は不要」でもありません。骨格まで作ってから、今の自分の用途と噛み合わないことに気づき、「今は使わない」を選んだ記録です。廃止ではありません。必要になれば戻れるように、骨格はそのまま残してあります。

姉妹編として、同じ日に書いた「[X の複数アカウント運用: Chrome 拡張で挫折 → Firefox Multi-Account Containers に流れ着いた記録](https://zenn.dev/ishizakahiroshi/articles/20260702-x-multi-account-firefox-container)」があります。そちらは「自作で境界突破は勝てない」の話で、この記事は「自作で情報蓄積も勝てない」の話です。同じ日に、別の場所で、似た形の白旗を上げました。

![](/images/20260702-declined-obsidian-for-now_infographic.png)

## 何を溜めれば正解か、決まりませんでした

まず、そこで詰まりました。

セッション振り返り、技術メモ、記事ネタ、家族の情報、顧客対応の教訓。どれも「溜めておけば将来役に立ちそう」に見えます。しかし全部溜めると、context が肥大化するのは自分の頭の方でした。分類コストが月末に効いてきて、結局「今週の分は来週の自分がやる」で 3 週間放置。

Obsidian のフォルダ・タグ・プロパティ設計で 1 日消える、というよく聞くトラップに片足を突っ込みかけました。骨格を作った勢いのまま「よし、今日は分類ルールを完成させよう」と考え、その日の夜には YAML プロパティを 12 個まで増やしていました。目的が入れ替わっていました。

ツールを整える快感は、そのまま「今日書かなくていい理由」に変わります。整える作業は具体的で、成果物も見えて、達成感も出やすい。書く作業は面倒で、成果物は地味で、達成感は薄い。天秤にかけたら整える方に流れるに決まっていました。

12 個目のプロパティは「confidence」でした。そのメモをどれくらい信頼して良いかを 0 から 1 で入れる欄です。書きながら「これ、あとで誰が更新するんだ」と自分で気づきました。半年後の自分が、半年前に付けた 0.7 と 0.8 の差を、正しく見直せるはずがありません。「信頼度」を管理するために毎回考える時間のほうが、そのメモを引く時間より長い。

その夜、YAML の 12 個目から順に消していき、最後は Markdown 本文しか残りませんでした。そこまで削って、初めて「あれ、これって普通のテキストファイルでいいのでは」と思いました。ノートを閉じ、コーヒーを淹れ直して、この記事の下書きに切り替えました。

## 半年前のメモが、次に開いた時には別物になっている

情報の耐用年数を意識せずに溜めると、9 割ゴミの倉庫になります。ここ数ヶ月で身に染みました。

Claude Code、Codex、Cursor、Antigravity。機能・モデル・価格の 3 点セットが、数ヶ月単位で書き換わっていきます。半年前に丁寧にまとめた「Claude Code の hook 記法」のメモは、次に読み返した時にはもう別物でした。書式が変わり、フラグが増え、既定値が動きました。書いた時点では正しかった情報が、次に必要になったときには信じてはいけない情報に変わっています。

![](/images/20260702-declined-obsidian-for-now_fig1.png)

上の図は「蓄積型 PKM」と「引き寄せ型 PKM」を並べた対比です。蓄積型は倉庫にモノを置いて必要になったら棚を探す作りで、引き寄せ型は必要になったときにその場で外から取り寄せる作りです。情報の陳腐化が速い領域ほど、引き寄せ型のほうが結果として鮮度が高くなります。

技術記事や公式 docs のように「常に最新版がどこかに置いてある」情報は、自分で持たなくてもすぐ取り出せます。むしろ持たないほうが、古い情報を最新だと勘違いする事故を防げます。

## 貯めた md を確認させると、1 回で数万トークン消える

ここが一番、実利で効きました。

貯めた md ファイルを「一応確認して」と AI に読ませると、1 回で数千から数万トークン消えます。そのうち半分は陳腐化していて、しかも AI はそれを「事実」として扱って返答に混ぜてしまいます。古い設定と新しい設定を平気で合成して、動かないコマンドを提案してきます。

いっそ web 検索や公式 docs を都度引かせたほうが、精度も鮮度も高いことに気づきました。「自分の倉庫を先に見る」よりも「今日の一次情報を見る」ほうが、実務上は速いのです。倉庫があると、まず倉庫を見ようとしてしまいます。倉庫があること自体が、判断を歪めていました。

:::message
自分で溜めた情報を AI に読ませるときは、「その情報の耐用年数」を先に評価する癖を付けると、無駄なトークンが減ります。家族の名前は 10 年後も真実ですが、AI CLI の設定は半年で書き換わります。同じ「メモ」でも、意味が全然違います。
:::

## 耐用年数の長いものだけ、薄く持てばよかった

そこで気づきました。耐用年数の長い情報だけを、薄く持てば十分でした。

家族の名前・生年月日・アレルギー。会社のサーバー IP・SSH ポート・ドメインの対応。個人 OSS のリリースチャネル対応表。どれも 10 年後も基本は真実で、変わっても差分だけ書き換えれば済みます。呼び出しの頻度も明確で、「今この文脈で必要かどうか」の判定が単純です。

![](/images/20260702-declined-obsidian-for-now_fig2.png)

上の図は自分が実際に使っているマスタデータの配置です。会社まわりは共有ストレージ経由の CSV に、家族まわりは個人クラウド同期の CSV に、認証情報は暗号化前提の TOML に、それぞれ薄く置いてあります。全部合わせても数百行に収まります。

これらは既に手元にありました。溜めるまでもなく、必要に応じて呼び出す仕組みだけで動いていました。Obsidian に移す必要も、graph で結ぶ必要もありませんでした。

## マスタデータは CSV と TOML で足りていた

具体的には、こういう配置です。

- 会社系: 共有ストレージ同期の CSV。人・会社・サーバー・アプリの 4 ファイル
- 家族系: 個人クラウド同期の CSV。名簿と、縦持ちの事実。非公開
- 認証系: 暗号化前提の TOML。実値が要るときだけ Read

どれも AI から「必要になったときだけ」参照される作りにしてあります。常時ロードはしません。人物 ID や company ID で突き合わせるだけで、バックリンクも graph も要りませんでした。CSV の 1 行が 1 レコード。それで足りていました。

変更頻度が低く、構造が浅い情報は、CSV や TOML で十分です。Obsidian の graph 表示が真価を発揮するのは、用語間の関係が緻密に絡み合う領域で、家族の名前と会社サーバーの間には、絡ませたい関係がありませんでした。

## 最新は引き寄せてもらえばいい

![](/images/20260702-declined-obsidian-for-now_fig3.png)

上の図は情報の耐用年数のスペクトルです。左端に家族の名前、右端に AI CLI の細かい仕様、中間に OSS の設計判断や具体 API の使い方が並びます。右に行くほど「自分で持たずに引き寄せる」ほうが実利があります。

最新の情報は AI が web 検索や公式 docs から都度引きます。vault に「AI の feature を書き溜める」よりも、そのときに `WebFetch` するほうが確実で、しかも半年後の自分に対しても親切です。半年前の自分が書いたメモは半年後の自分にとってノイズになりがちですが、公式 docs は常に最新に更新されています。

検索の質は年々上がっています。個人 vault で AI 検索の代替をしようとするのは、家でラーメン屋を開くのに近い挑戦です。可能ではあります。ただ、外に食べに行ったほうが美味しくて速いのが実情です。

## Obsidian が刺さる人がいる。その人には正解

ここは、本気で書きます。

研究者や論文執筆者、長期プロジェクトを追い続ける人にとって、Obsidian のバックリンクと graph は思考の一部になります。用語間の関係が緻密に張り巡らされた領域では、リンクを辿ること自体が発想の呼び水になります。数年単位で参照される概念地図を持てる人には、これ以上の道具はなかなかありません。

「10 年書き続けるつもりの知識体系」を持っている人には、Obsidian は間違いなく正解の一つです。ローカルファースト、Markdown、プラグインの豊富さ、コミュニティの厚さ。どれも長期運用に向いた設計で、10 年後にも同じデータ形式で読める安心感があります。

具体的に想像してみます。ある言語学の研究者が、方言と語源と地域史のノートを 5 年書き続けているとして、A 地域の語彙メモから B 時代の史料メモへ、そこから C 方言との共通祖先の推定メモへ、リンクを辿って一晩過ごせる。この体験は、CSV のマスタデータでは絶対に出せません。graph をぼんやり眺めているうちに、隣接クラスタの意外な結節点に気づく、あの感覚。自分が持てなかったのは、そういう体験でした。

執筆家の場合も似ています。長編小説の登場人物の関係図、伏線の張り替え、章ごとの視点移動を追う。graph が思考の外部装置になります。ページ数が積み上がるほど道具の価値が増える構造で、これは典型的な蓄積型が勝つ領域です。

自分にはそういう体系がありませんでした。技術は移り変わり、業務は案件で変わり、興味の中心も年単位で動きます。10 年物の一貫した知識体系を書き続ける準備が、今の自分にはなかっただけです。準備がないのに 10 年の道具を先に整えると、道具に人生を合わせる方向に引っ張られます。それは自分がやりたいことではありませんでした。

Obsidian が悪いのではありません。自分の用途と噛み合わなかっただけです。この違いは大きいので、読んでいる方の中で Obsidian を愛用している方がいたら、そのまま続けてください。あなたの用途では、それが最適です。

## 今は使わない、でも骨格は残す

![](/images/20260702-declined-obsidian-for-now_fig4.png)

上の図は「あなたに Obsidian は要るか？」という、ざっくりの判断フローチャートです。5 年以上追い続ける知識体系があるか、用語間の関係を辿りたいか、と 2 問だけ聞きます。両方 Yes ならたぶん正解、片方だけなら要検討、両方 No なら薄いマスタデータで足ります。

自分の vault plan は「保留」に置きました。廃止ではありません。骨格・ディレクトリ構造・plan md は、そのまま残しています。将来の自分が「あの時考えたことをそのまま拾える」ように、消さずに寝かせてあります。

今の用途が変わったら、たとえば長期の研究テーマや連続執筆プロジェクトを持ったら、そのときに骨格から再開します。セッションの語りのようなものは、当面プロジェクト内の `docs/local/` に個人メモとして残します。横断集約はしません。公開できる知見は、まず Zenn や note に記事として出します。他人にも役立ちますし、自分の retrieval にもなります。マスタデータは CSV と TOML で薄く保つ、既存の集約先を維持します。

## 用途と情報鮮度で、PKM の正解は変わる

蓄積型が正解の人もいれば、引き寄せ型が正解の人もいます。今の自分は薄い + 即時性重視で、蓄積は要りませんでした。でも 5 年後に「10 年書き続けるつもりの知識体系」ができたら、Obsidian に戻る可能性は普通にあります。骨格を残しておいたのは、そのためです。

Obsidian を使う人、使わない人、今は使わない人。どれも自分の用途に合った選択をしています。

迷っている方には、vault の骨格を作ってみて、1 週間運用してみて、続かないなら「保留」に置くことをおすすめします。骨格作成は 30 分、保留判断は 1 分です。「廃止」しなくていい、というのが、たぶん今日一番書きたかったことです。

小さく残して、必要になれば戻る。それだけを、これからも続けていきます。

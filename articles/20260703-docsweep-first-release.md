手元の管理対象を数えたら、25 プロジェクトに 408 個ありました。plan_*.md、bugfix_*.md、pending_*.md。AI コーディングツールに「計画を md に残して」と頼んだ結果が、この山です。

この山を捌く道具 docsweep を作って、今日 PyPI に初リリースしました。この記事はそのツール紹介と、リリースまでの 12 日で起きたことの話です。

## AI は md を書くのが好きすぎる

Claude Code や Codex と仕事をしていると、作業記録の md がどんどん増えます。計画は plan、障害対応は bugfix、保留は pending。ここまでは良い習慣だと思っています。書くこと自体はコストが低いし、次のセッションの AI が文脈を拾えます。

問題は、その後です。

- 実装が終わっているのに、plan のラベルが「実行中」のまま放置される
- 3 週間前の「様子見」が、様子を見られないまま化石になる
- 新しいセッションの AI が古い plan を拾って、終わった作業をやり直そうとする

つい先日も、とっくに実装が終わった plan が 2 本、「実行中」のまま期日切れで吊るされているのを見つけました。コードと突き合わせたら完了条件を全部満たしている。ラベルを書き換え忘れていただけでした。

書く仕組みはあるのに、畳む仕組みがない。408 個はその結果です。

![AI セッションごとに md が増えて腐っていく流れ](/images/20260703-docsweep-first-release-fig1.png)

上の図が今回の困りごとの全体像です。生成は毎セッション起きるのに、状態の更新と片付けは人間の記憶頼みなので、時間とともに「実態と合わない md」が溜まっていきます。

## 既存の道具では何が足りなかったか

タスク管理ツールに寄せる手はあります。ただ、AI との共同作業では「md ファイルそのものが正本」であることに価値があります。ツールのデータベースに移した瞬間、AI がファイルとして読めなくなり、git 履歴からも切り離されます。

Obsidian のようなノートアプリも考えました。でも欲しいのはリンクやグラフではなく、「これはもう終わったか？」への答えです。

つまり、md はそのまま、状態だけを機械的に扱いたい。それなら作るしかない、となりました。

## 設計の芯は「ラベルを立てるのは人、運ぶのは機械」

docsweep の仕組みは単純です。md の H1 先頭に付けたステータスラベルを機械的に読み取ります。

```markdown
# [実行中] 認証フローのリファクタ
```

状態は一直線に流れます。

```
plan:   [保留] → [計画] → [実行中] → [様子見] → [完了]
bugfix:          [対応中] → [様子見] → [完了]
（どの状態からでも [廃止] へ分岐できる）
```

![状態モデルの遷移図（様子見は守られる）](/images/20260703-docsweep-first-release-fig2.png)

図の通り、「完了」と「廃止」だけが archive 送りの対象です。設計で一番こだわったのはここで、判断と作業を分離しています。

- ラベルを立てる（判断）: 人か AI が明示的にやる
- archive へ運ぶ（作業）: docsweep が自動でやる
- 「様子見」は絶対に自動移送しない: 直したけど再発を見たい、という寝かせ期間を機械から守る

期日が切れても、勝手に「廃止」にはしません。候補として赤く吊るすだけ。最後のワンクリックは必ず人の手に残しています。物理削除の口も実装として持っていません。最悪でも archive 止まりで、復元できます。

:::message
「AI に任せて自動で掃除」ではなく「AI と人が判断しやすい形に、機械が下ごしらえする」が docsweep の立ち位置です。
:::

## CLI と看板とMCP、三つの顔で同じ一つの口を叩く

使い口は 3 つ用意しました。

```bash
pip install 'docsweep[all]'

python -m docsweep triage          # 残作業を古い順に
python -m docsweep sweep --dry-run # archive 候補の下見
python -m docsweep serve --root ~/dev  # Web UI（看板）
python -m docsweep mcp             # AI エージェント向け MCP サーバー
```

![CLI / Web UI / MCP が共通の services 層を叩く構成図](/images/20260703-docsweep-first-release-fig3.png)

構成は上の図の通りで、CLI も Web UI も MCP も、ファイルを書き換える処理は同じ services 層を通ります。Web UI にだけ特別な権限がある、みたいな抜け道を作らないためです。

Web UI は看板（カンバン）方式にしました。「やり忘れ」「今日」「実行中」の 3 列に、期日ベースでカードが並びます。朝これを開いて、溜まったカードを一括で「+1 週」に先送りしたり、まとめて廃止したりします。

![看板ボードのレイアウト概念図（3 列 + 一括編集バー）](/images/20260703-docsweep-first-release-fig4.png)

看板の上部には複数選択の一括操作バーがあり、列単位・画面全体の全選択もできます。「今日は全部諦めて 1 週間送る」が 2 クリックで終わるのは、想像以上に精神衛生に効きました。

MCP 経由なら AI から直接叩けます。セッション開始時に triage を読ませる導線を注入する機能もあって、AI が「先週のこの plan、実装済みっぽいのでラベル更新しますか」と聞いてくる世界になります。

## ドッグフード 2 日のつもりが 12 日になった

v0.1.0 は 6 月 21 日に出すつもりでした。「2 日ほど自分で使ってから」と決めてタグを打たずにいたら、出てくる出てくる。結局 12 日使い込んで、その間に看板 UI の全面改修、一括編集、Undo、frontmatter 対応が入りました。予定は崩れましたが、初版としては明らかに良いものになったので、これは崩れて正解でした。

一番ヒヤッとしたのはリリース当日です。自分のリポジトリで docsweep の sweep を走らせたら、archive の移送先がリポジトリ直下の archive/ でした。そこ、gitignore されていない場所です。つまり非公開の作業ログが、公開リポジトリの追跡対象に移動する寸前でした。

dry-run で下見していたので実害はゼロ。設定ファイルで移送先を gitignore 済みの場所へ変えて、ついでに「設定がフラグなしだと読まれない」というバグも見つけて直しました。sweep は複数プロジェクトを横断するのに、archive 先の解決が起動時の単一設定しか見ていなかった。移送の直前に対象プロジェクト自身の設定を読むように変えています。

![archive 移送先バグの before/after](/images/20260703-docsweep-first-release-fig5.png)

上の図がその修正の前後です。自分のツールの危うさを自分のワークフローで踏むのは、ドッグフーディングの一番おいしいところだと思います。リリース前に踏めてよかった。

もうひとつ、リリース直前に思い切った撤去もやりました。Web UI の一部が TypeScript + Vite でビルドされていたのですが、看板 UI への全面移行で旧画面が不要になり、ビルドスタックごと削除。Python パッケージなのに Node 環境が要る、という二重スタックが消えて、開発も配布も Python だけで完結するようになりました。

![12 日間で変わったことの年表](/images/20260703-docsweep-first-release-fig6.png)

12 日の変遷を並べたのが上の図です。「2 日のドッグフード」の見積もりは 6 倍に膨れましたが、腐る前の初版に気づきを全部畳み込めたと思えば安いものです。

## 使ってみたい方へ

```bash
pip install 'docsweep[all]'
python -m docsweep triage
```

リポジトリはこちらです。

https://github.com/ishizakahiroshi/docsweep

H1 ラベルの規約はテンプレートとして同梱していて、`python -m docsweep inject` で自分のプロジェクトの CLAUDE.md / AGENTS.md に注入できます。Claude Code でも Codex でも、AI に「この規約で md を書いて」と伝わる形です。

frontmatter は OKF（Open Knowledge Format）互換にしてあるので、docsweep をやめても md は腐りません。やめる自由があるツールの方が、安心して始められると思っています。

## 次の一歩

リリースした夜に、自分の triage を叩いたら残作業ゼロでした。408 個あった山は、要判断も期日切れも空。この状態を保てるかどうかが、道具の本当の審判になります。

まずは毎朝、看板を開くところから。地味な習慣を、これからも続けていきます。

朝、Nextcloud の管理画面を開いたら、ダッシュボードが真っ黒な「このページは動作していません」に変わっていました。HTTP 500。トップも設定もファイル一覧も、どこを開いても 500。前日まで普通に動いていたものが、一晩で全滅していました。

きっかけは、はっきりしていました。直前にアプリを一つ入れた。だから最初は「入れたアプリが悪いんだろう」と思ったんです。でも、犯人はアプリではありませんでした。PHP 8.3 の JIT でした。

この記事は、その 500 を切り分けて潰すまでの記録です。前半で「JIT と opcache って何が違うの」を噛み砕いて、後半で実際に踏んだバグの切り分けを順番に書きます。同じエラーメッセージで検索してたどり着いた人が、遠回りせずに済むように。

![](/images/2026-07-07_php83-jit-preg-match-bug_hero.png)

![](/images/2026-07-07_php83-jit-preg-match-bug_infographic_v2.png)

## 「アプリを入れた直後」だったので、アプリを疑った

最初の見立ては素直なものでした。アプリを入れた直後に落ちたのだから、そのアプリのコードがどこかで例外を投げているに違いない、と。

でもこの見立て、後から振り返ると半分正解で半分外れでした。「アプリを入れた」という事実は確かに引き金だったのですが、落ちた原因はアプリのコードではなかった。そこは切り分けを進めないと見えてきませんでした。

まずログを読みます。Nextcloud なら `data/nextcloud.log` です。そこにこんな行が並んでいました。

:::details 実際に出ていたエラー（抜粋）

```
TypeError: preg_match(): Argument #2 ($subject) must be of type string,
string given
  at apps/theming/lib/ThemingDefaults.php#187
```

:::

読んだ瞬間、少し固まりました。「string 型でなければならないのに、string が渡された」。日本語にすると意味が通りません。string を渡してるのに string 型じゃないと怒られている。自己矛盾です。

この一文が、実は今回の全部を説明する鍵でした。

## 値が壊れているのかと思ったら、そもそも空だった

`preg_match($pattern, $subject)` の第 2 引数が変だと言われている。ということは、そこに渡している変数の中身がおかしいのかもしれない。該当行は、テーマのメインカラーを取り出して色コードの正規表現にかけている、ごく普通のコードでした。

```php
// $userPrimaryColor は文字列が入るはずの変数
if (preg_match('/^\#([0-9a-f]{3}|[0-9a-f]{6})$/i', $userPrimaryColor)) {
    return $userPrimaryColor;
}
```

じゃあこの色設定の値が壊れているのか、と DB を見にいきました。ところが、テーマのユーザー設定は一件も入っていない。空でした。

ここで手が止まります。設定が空なら、この変数には空文字が入るだけです。空文字を `preg_match` に渡しても、マッチしないので `0` が返るだけで、エラーになんてなりません。値は壊れていない。なのにコードは落ちる。

「値が壊れているせい」という線が、ここで消えました。値が正しいのにコードが落ちるなら、疑うべきは値ではなく、コードを動かしている土台の方です。

## 決め手は「設定を何も変えずに再起動しただけで直った」

土台、つまり PHP の設定を見にいくと、`opcache.jit` が有効になっていました。しかも tracing モードでガッツリ効かせる設定。ここで嫌な予感がしました。

そこで、切り分けとして一番きれいな一手を打ちます。JIT の設定は一切いじらない。ただ PHP-FPM を再起動するだけ。

```bash
sudo systemctl restart php8.3-fpm
```

これだけで、全ページが 200 に戻りました。設定は何も変えていないのに、です。

:::message
設定を変えずに再起動しただけで直った、というのが決定的でした。「コード自体は正しい。壊れていたのは、実行するために生成された機械語のほうだ」と、これで確定できます。
:::

PHP-FPM の再起動は opcache をまるごとクリアします。クリアされた直後は、コードが正しく再コンパイルされる。だから直る。値でもコードでもなく、コンパイル結果が壊れていた。犯人は JIT でした。

## そもそも JIT って何をしているのか

ここで一度、後半の切り分けを止めて前半の解説に戻ります。JIT と opcache がごっちゃになっていると、この後の判断が飲み込みにくいので。

PHP は普通、コードを一行ずつその場で解釈しながら動きます。書き換えに強い代わりに、同じ処理を何度も通ると毎回解釈し直すので少し遅い。

そこで二段のキャッシュが乗っています。

- **opcache**: PHP コードを一度コンパイルしたバイトコードを保存して使い回す。パースし直す手間を省く。速度に効くのは主にこれ
- **JIT**: そのバイトコードを、さらに CPU が直接読める機械語まで翻訳しておく。opcache の上に乗る「おまけ」の高速化

![](/images/2026-07-07_php83-jit-preg-match-bug_fig1.png)

上の図のように、opcache が「バイトコードの使い回し」で、JIT はその先の「機械語への翻訳」です。今回壊れたのは JIT の翻訳結果で、opcache 本体ではありませんでした。

ここで、二つの役割の違いをもう少しはっきりさせておきます。名前が似ているので混同しがちですが、効く場面も、入れるべきかどうかも、まったく違います。

**opcache は「毎回の下ごしらえ」を省くためのもの**です。PHP はリクエストが来るたびにソースを読み、パースして、バイトコードにコンパイルします。この下ごしらえは、同じファイルなら毎回まったく同じ結果になる。なのに毎回やり直すのは無駄なので、一度作ったバイトコードを保存して使い回す。これが opcache です。どんな PHP アプリでも確実に効くので、Nextcloud も PHP 本体も「本番では有効にしてください」と案内している、いわば定番装備です。

**JIT は「特定の重い計算」を速くするためのもの**です。バイトコードをさらに CPU 直読みの機械語まで翻訳しておき、実行時のひと手間を消します。ただしこれが効くのは、CPU がひたすら数字をこね回す処理。画像のピクセル演算、科学計算、暗号のループのような、いわゆる CPU バウンドな処理です。逆に言うと、そういう処理を持たないアプリではほとんど出番がありません。

一言でたとえると、opcache は「毎回作る下ごしらえを冷蔵庫に保存して使い回す」仕組み、JIT は「その料理をさらに現場の言葉に同時通訳する」仕組みです。同時通訳は速い。でも通訳者（JIT）が誤訳すると、今回のように現場が止まります。

![](/images/2026-07-07_php83-jit-preg-match-bug_fig2.png)

上の図は、その opcache と JIT を「いつ効くか・入れるべきか・今回どうだったか」で並べたものです。opcache は全アプリで効く必須級、JIT は計算処理向けの任意装備、という温度差が今回の判断の土台になりました。

問題は、この「機械語に翻訳するプログラム」自体にバグがあった場合です。翻訳結果の機械語が間違ったものになる。今回はまさにそれで、色を取り出す関数を翻訳したときに引数の型の扱いを取り違え、「string を渡したのに string 型でない」というあり得ない機械語ができあがっていました。

普通の解釈実行なら絶対に起きません。実際、JIT を切ったら一発で消えました。値ではなく、実行コードそのものが壊れていた、という証拠です。

## なぜ「アプリを入れた直後」に出たのか

ここで最初の「アプリを入れた直後」に話が戻ります。

JIT が機械語を作り直すのは、opcache がリセットされてコードを再コンパイルするときです。そして Nextcloud は、アプリを入れる・有効化する・更新するたびに opcache をリセットします。読み込むコードが変わるので、古いキャッシュを捨てるためです。

![](/images/2026-07-07_php83-jit-preg-match-bug_fig3.png)

図にすると、こういう連鎖でした。アプリ導入が opcache リセットを呼び、それが JIT の再コンパイルを走らせ、そこで誤った機械語ができて、全ページが 500 になる。アプリのコードは一度も登場しません。ただ再コンパイルの引き金を引いただけです。

だから「アプリを入れた直後に必ず落ちる」ように見えていた。でも中身は、どのアプリでも同じでした。opcache をリセットするものなら何でも引き金になる。アプリは悪くなかったんです。

## JIT は切ってよかったのか

原因はわかった。では対処です。素直に JIT を無効化しました。

```ini
opcache.jit=off
```

無効化して再起動。以降、どのアプリを入れても 500 は再発しなくなりました。

ここで一つ、立ち止まって考えたいことがあります。「JIT って高速化のための機能でしょう。切っちゃって遅くならないの?」という不安です。結論から言うと、この用途では切って問題ありませんでした。理由を三つ書いておきます。

まず、**opcache と JIT は別物**だということ。切ったのは JIT だけで、速度に効く opcache 本体は有効なままです。バイトコードの使い回しは生きているので、キャッシュを全部捨てたわけではありません。

次に、**JIT が効くのは CPU をぶん回す処理**だということ。画像のピクセル演算や数値計算のような、CPU が延々ループする処理では速くなります。でも Nextcloud のような Web アプリは、処理時間の大半が DB アクセスやファイル I/O やネットワーク待ちで、CPU 計算はごく一部です。待ち時間が支配的なところに JIT を足しても、体感はほとんど変わりません。

最後に、**そもそも要求されていない**こと。Nextcloud の管理ドキュメントが「入れてください」と挙げている PHP 設定は opcache 系だけで、`opcache.jit` は推奨リストに入っていません。動作要件として求められているわけではないんです。

効果が薄いのに、今回みたいな型エラーを撒く。だとしたら、Web 用途では入れないほうが無難、という位置づけになります。実際に踏んでみて、その通りだと納得しました。

## これ、本当に PHP のバグなの?（一次情報の裏取り）

ここは正直に線を引いておきます。今回こちらで確実に言えるのは「この環境では JIT を切ったら直った、だから JIT の誤コンパイルが原因」までです。「PHP のバグ番号 XXX が原因」と 1 件に同定したわけではありません。実測と、公式バグとしての同定は別物なので、裏取りした一次情報を置いておきます。

まず、今回のような「string を渡したのに string 型じゃない」という自己矛盾したメッセージは、私の環境だけの珍現象ではありませんでした。PHP 本体の issue にも、Nextcloud のコードで型宣言された戻り値を組み込み関数に渡したときに、まったく同じ形のエラー（`must be of type string, string given`）が出た報告があります（こちらは `strtolower` 版）。値ではなく型情報そのものが壊れると、こういう意味の通らないメッセージになります。

そして、その「型情報を壊す」側の犯人として tracing JIT が名指しされているバグも複数あります。tracing JIT が未定義値の型を取り違えるものなどで、いずれも `opcache.jit` を無効化すると回避できるという点で、今回と地続きです。

整理すると、立ち位置はこうなります。

- 「自己矛盾メッセージ＝型情報の破損」は既知のパターン
- 「tracing JIT が型情報を壊す」バグも PHP に複数報告あり
- 今回は実測（JIT off で解消）で、この環境では JIT が原因と確定

この三つが揃っている、というのが正確なところです。「PHP 8.3 の JIT のバグだと断言」ではなく、「実測で JIT が原因と確定し、PHP 側にも同系統のバグ報告がある」。ここは分けて捉えておくのが誠実だと思っています。同じ症状に当たった人は、まず自分の環境で `opcache.jit=off` にして再起動し、直るかどうかを確かめるのが一番速い切り分けです。

## 学んだこと

- **「値が壊れている」と「コードが落ちる」は別物**。設定が空なのに落ちたなら、疑う先は値ではなく実行環境。ここで一段ずらせたのが分かれ目でした
- **自己矛盾したエラーメッセージは、実行コードの破損を疑う**。「string を渡したのに string 型じゃない」のような日本語にして意味が通らないメッセージは、値やロジックではなく、JIT やコンパイラ側の異常を示すことがあります
- **設定を変えずに再起動して直るかどうか**は、強力な切り分けになる。直れば「コードは正しい・キャッシュ側が壊れていた」と一発で分かります
- **「直後に起きた」を「それが原因」と混同しない**。引き金と原因は別。アプリ導入は引き金で、原因は JIT でした

## 締め

派手な結論はありません。ただ、朝いちで全ページ 500 を見たときに「アプリのせいだ」で止まらず、「設定を変えずに再起動したら直るか」を一手はさめたのは、今回よかったと思っています。

自己矛盾したエラーを見たら、値やロジックを疑う前に、足元のキャッシュを一度疑う。この小さな順番を、これからも忘れないようにしておきます。

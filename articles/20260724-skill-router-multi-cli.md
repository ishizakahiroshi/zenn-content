登録してあるスキルが30個を超えたあたりで、自分でも名前を思い出せなくなりました。

AIコーディングCLI（Claude Code や Codex など）には、定型作業を手順書にした「スキル」を仕込めます。記事を書く、アイコンを作る、リリースする。便利なので増やしていったら、いざ使うときに「あれ、記事書くやつって何て名前だっけ」と、自分の作ったものを探す羽目になりました。

毎回スキル名を正確に打つのが、地味に面倒だったんです。

## 結局これをやる

先に結論から書きます。やったのは、**「今の話の流れに合いそうなスキルを、近さと根拠つきの番号リストで出して、選ぶだけ」**にするルーターを1本作ったことです。

こんな感じで動きます。

```
> S?
今の状況に合いそうなスキル:
1. write-article（高）： 「記事にしたい」が起動語と一致
2. session-recap（中）： 作業が一区切りした文脈に合う
3. publish-article（低）： 「記事」は共通だが公開段階ではない
番号を選ぶと起動します。
```

`S?` と打つだけ。あとは番号を選ぶ。名前を思い出す必要はありません。

![記事の要約](/images/skill-router_infographic.png)

## なぜ「通知に任せる」ではダメだったのか

最初は「AIが文脈を読んで、勝手に良いスキルを提案してくれればいいのでは」と考えていました。

実際、多くのCLIには「今このスキルが使えますよ」という候補通知の仕組みがあります。ただ、これが毎ターン機械的に絞り込まれていて、取りこぼす。過去に「HTMLで説明して」と頼んだのに、HTML作成用のスキルが通知に出てこず、AIがゼロからHTMLを書き始めたことがありました。

通知は当てにならない。かといって、毎ターン「候補を照合しろ」という命令を差し込むフックを噛ませると、今度は無関係な入力でも判定が走って、文脈がノイズで太る。

だから、**能動的に「S?」と一言打った時だけ**候補を出す、という半自動に落ち着きました。打つ手間はほぼゼロだし、要らないときは黙っている。ちょうどいい塩梅でした。

## 精密そうな「適合度の数字」をやめた話

最初は候補に、いかにも精密そうなパーセントの数字を振っていました。でも、書いていて気づいたんです。これ、計算してない。

中身は拍子抜けするほど素朴で、スキルの索引（起動語と一言説明が並んだ表）を読んで、直近の会話と意味的に突き合わせているだけ。点数を出す関数なんてどこにもなくて、AIが「近いな」「遠いな」と感覚で判断しているだけです。それを2桁の数字にすると、実際より正確そうに見えてしまう。むしろ嘘くさい。

なので、数字はやめました。今はこの3段階です。

- 高: 起動語がほぼそのまま入っている、またはほぼ言い換え
- 中: 意味は合うが言葉が違う
- 低: かすっている程度。これより下は出さない

そして**必ず「なぜそう思ったか」を一行添える**。「『記事にしたい』が起動語と一致」みたいに。数字を消して根拠を出すようにしたら、そのほうがよほど納得して選べるようになりました。

![](/images/skill-router_fig1.png)

上の図のように、段階はざっくり3つ。数字の代わりに「なぜ合うと思ったか」を必ず添える、というだけの単純な作りです。

「低」より下を切っているのは、候補を水増しすると選ぶ側が疲れるからです。「一応これも…」を並べられると、結局全部読み直すことになる。上位2〜5件に絞る、という制約が地味に効きました。

## 一番こだわったのは「全CLIで同じように動く」こと

自分は Claude Code だけでなく、Codex、Cursor、Grok、Copilot と、複数のCLIを行き来しています。だから「Claude専用の便利機能」を作っても、隣のCLIに移った瞬間に使えなくなるのが一番困る。

ここで効いたのが、**スキルの置き場を1箇所にして、各CLIにはジャンクション（ディレクトリの別名リンク）で同じ棚を見せている**という、前からやっていた構成でした。

```
skills/            ← 実体はここ1箇所
  write-article/
  make-icon/
  skill-router/    ← 今回追加したルーター
  ...
~/.claude/skills   → 上のフォルダへのジャンクション
~/.agents/skills   → 同上（Codex 用）
~/.cursor/skills   → 同上
...
```

![](/images/skill-router_illustration-1.png)

棚は1つ、そこから何本もの線が各CLIへ伸びている、というのがこの構成の実感です。本を1冊足せば、全部の場所に同時に並ぶ。

おかげで、ルーターの本体（`SKILL.md` 1ファイル）を棚に置くだけで、全部のCLIに一斉に伝わりました。CLIごとに書き分ける必要がない。追加したあと、5つのパス全部でファイルが見えるか確認して、それで終わりです。

:::message
各CLIは「スキルの説明文」を頼りに起動語を拾います。`S?` だけだと薄いので、「どのスキル」「おすすめスキル」「スキル候補」あたりの自然な言い回しも起動語に厚めに入れておくと拾われやすくなります。Claude Code 側は `/S ?` という短い入口を用意して、そこからこのルーターへ委譲しています。
:::

実体は1つ、入口は各CLIの作法に合わせる、という形です。

## 作ってみて思ったこと

大した実装ではありません。やったことは「索引を読んで、近さと根拠を出して、番号で選ばせる」だけ。新しいライブラリも、フックも使っていない。

それでも、`S?` の3ストロークで済むようになっただけで、スキルを増やすことへの心理的なハードルがだいぶ下がりました。「名前を覚えていないと呼び出せない」という縛りが消えると、道具は素直に増やしていける。

まだ近さの判断は勘に頼っているし、上位を何件まで出すかも、しばらく使いながら直すつもりです。完璧な採点ロジックにするより、「だいたい合ってて、外したら番号で選び直せる」くらいの緩さがちょうどいい気もしています。

道具を作るより、道具を思い出すコストの方を先に潰す。そういう小さい手当てを、これからも続けていきます。
